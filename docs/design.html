Scuttledeck

An open-source control plane for the Claude Code GitHub Action: which repos run it, what it's doing on your PRs, what it costs — one dashboard for the whole fleet.

01 · Problem

The gap: agents are running CI, and nobody can see the fleet

Teams are wiring anthropics/claude-code-action into dozens of repos — @claude mentions, automated PR review, scheduled maintenance jobs. Each run is an autonomous agent spending real tokens and writing real code. Today the only way to answer basic operational questions is to click through GitHub's Actions tab repo by repo:

  • Adoption — which of our 200 repos actually have the action installed, and on which triggers?
  • Activity — how many PRs did Claude review this week? How many did it open? What's the failure rate?
  • Cost — what did agentic CI cost this month, by repo, by workflow, by model? What does one PR review cost?
  • Health — which workflows are failing, timing out, or burning tokens abnormally?

Nothing on the market answers these together. Anthropic ships no UI for the action (GitHub is the UI). The Anthropic Console shows org-level spend but knows nothing about repos or PRs. Existing community dashboards (claudecodeui, claude-code-webui, Agent Monitor) target local CLI sessions, not CI. Datadog/Honeycomb can receive Claude Code's OpenTelemetry metrics, but that's a build-it-yourself dashboard with no GitHub context and no turnkey story.

Thesis: the winning shape is a self-hostable GitHub App + telemetry collector that joins three data planes — GitHub events, Claude Code OpenTelemetry, and Anthropic Admin APIs — into one PR-centric, cost-aware fleet view. Think "Codecov for agentic CI."

02 · Users

Who it serves, and the question each one asks

Platform / DevEx engineer PRIMARY

Owns the org-wide rollout. Asks: "Where is the action installed, is it healthy, and which workflow configs drift from our golden template?" Needs fleet inventory, failure alerts, config visibility.

Engineering manager

Justifies the spend. Asks: "How many PRs is Claude actually reviewing or authoring, and is that number growing?" Needs adoption trends, per-team activity, PR outcomes.

FinOps / budget owner

Watches the bill. Asks: "What does this cost per repo and per PR, and warn me before a runaway loop burns $500." Needs cost breakdowns, unit economics, budget alerts.

Security engineer

Audits the agent. Asks: "Which runs had write access, what did they change, and who triggered them?" Needs an audit trail of triggers, actors, and outputs per run.

03 · Data sources

What can actually be measured (verified)

This design is grounded in what the platforms expose today. The single most important finding: the action itself exposes no per-run cost or token output — its outputs are only conclusion, session_id, and an optional user-schema structured_output. So per-run economics must come from telemetry or Anthropic's Admin APIs, and joining those to GitHub runs is the core technical work of this product.

SourceWhat it givesGranularityConstraint
GitHub App webhooks workflow_run, workflow_job, pull_request, pull_request_review, issue_comment, installation_repositories, push — real-time run status, durations, Claude-bot comments/reviews Per event, real time No cost data; needs app installed org-wide
GitHub REST (backfill) Repo list → workflow files → parse YAML for uses: anthropics/claude-code-action; historical runs Batch scan No API filter for "workflows using action X" — must fetch & parse YAML. 5k req/hr; needs ETag caching
Claude Code OTel claude_code.token.usage, claude_code.cost.usage, session/tool metrics via CLAUDE_CODE_ENABLE_TELEMETRY=1 + OTLP env vars Per run, near-real-time (60s batches) Opt-in: each workflow must set env vars; needs a reachable OTLP endpoint
Claude Code Analytics API /v1/organizations/usage_report/claude_code — sessions, LoC, commits, PRs created, per-model tokens + estimated cost, attributed to api_actor (API key name) Per actor per day (~1h freshness) Requires Admin API key; daily aggregates, not per run
Usage & Cost Admin API usage_report/messages (group by api_key/model, 1m–1d buckets) and cost_report (billing-accurate USD, daily) 1 min – 1 day buckets Admin API key; no repo/run context — reconciliation only

Auth-mode gap: when the action authenticates with a Claude subscription OAuth token instead of an API key, there is no Admin-API cost trail — usage is covered by the subscription. For those installs Scuttledeck shows token counts from OTel and labels cost "included in subscription." This must be a first-class state in the UI, not an error.

Components

ComponentResponsibility
GitHub AppInstalled org-wide with least privilege: actions:read, contents:read, pull_requests:read, metadata:read. Delivers webhooks; its installation token powers the backfill scanner. One app, many installations → multi-org from day one.
Ingest serviceTwo public endpoints. /webhooks/github: verifies HMAC signature, enqueues raw event, returns 200 fast. /v1/otlp/metrics: accepts OTLP/HTTP from CI runners, authenticated by per-install ingest tokens; extracts token/cost counters with their resource attributes.
Discovery scannerNightly (and on push to workflow files): list repos → list workflows → fetch YAML via contents API with ETags → parse for claude-code-action refs, triggers, model config. Produces the fleet inventory and flags version drift (e.g. repos pinned to an old action version).
CorrelatorThe heart of the product: joins an OTel session to its GitHub run via resource attributes (run id injected at workflow level — see §05), and joins runs to PRs via the workflow_run payload and Claude-bot comment/review events. Falls back to time+repo heuristics when attributes are missing, marked lower-confidence.
Anthropic pollerHourly pull of Claude Code Analytics (api_actor-attributed sessions, tokens, estimated cost) and daily pull of cost_report for billing-accurate reconciliation.
Alert engineRules evaluated on ingest and on schedule: monthly budget per org/repo, cost-per-run anomaly (>N× trailing median), failure-rate spike, "action inactive in repo X for 14 days." Notifies Slack webhook / email.
05 · Cost attribution

Three tiers, graceful degradation

Cost answers get better the more the customer opts in. Each tier works without the next; the dashboard labels the provenance of every dollar figure.

Tier 1 — zero workflow changes MVP

Customer supplies an Anthropic Admin API key. The poller pulls the Claude Code Analytics API daily: per-API-key sessions, tokens, estimated cost, plus commits/PRs counted by Anthropic. If teams follow the recommended convention of one API key per repo or team (e.g. gh-action-payments-api), this alone yields per-repo cost with no CI changes. Accuracy: daily, estimated.

Tier 2 — per-run precision via OTel FLAGSHIP

Ship a one-line companion step, scuttledeck/setup@v1, dropped in before the action:

- uses: scuttledeck/setup@v1
  with:
    endpoint: https://scuttledeck.internal.acme.dev
    token: ${{ secrets.SCUTTLEDECK_TOKEN }}
- uses: anthropics/claude-code-action@v1
  ...

The setup step exports CLAUDE_CODE_ENABLE_TELEMETRY=1, OTEL_METRICS_EXPORTER=otlp, the OTLP endpoint + auth header, and — critically — OTEL_RESOURCE_ATTRIBUTES carrying github.repo, github.run_id, github.workflow, and PR number. Every token/cost counter now lands pre-joined to its exact run. Accuracy: per run, ~60s latency.

Tier 3 — billing truth

Daily cost_report reconciliation: Tier 1/2 numbers are estimates; the cost report is what Anthropic bills. Scuttledeck scales estimated figures to the reconciled total and shows drift ("estimates within 3% of invoice this month"). Subscription-OAuth installs skip cost and report tokens only.

Unit economics fall out of the join: cost per PR review = session cost of runs attached to that PR; cost per merged agent PR, cost per repo per week, tokens per turn follow the same path.

08 · Security & privacy

Hold secrets, not prompts

  • Least-privilege GitHub App — read-only scopes only; Scuttledeck never writes to repos. Webhook HMAC verified on every delivery.
  • No prompt or code content stored. Claude Code's OTel omits prompt/tool content by default; Scuttledeck never sets the opt-in flags and drops any log bodies at ingest. Metadata only: counts, costs, durations, statuses, PR numbers.
  • Secrets — Admin API key and app private key encrypted at rest (libsodium sealed box), decrypted only in the poller; ingest tokens stored hashed, scoped per installation, revocable.
  • Self-hosted by default — telemetry stays inside the customer network; the OTLP endpoint needs to be reachable from CI runners only. SSO (OIDC) + role-based read access for the dashboard.
09 · Roadmap

Four phases to a fleet product

  1. Spike — prove the joins ~1 wk
    GitHub App against one test org: webhook ingest, YAML discovery scan, and an OTel export from a real claude-code-action run landing in Postgres with run-id attributes intact. Kills the riskiest assumptions before any UI exists.
    EXIT: one run visible end-to-end with its true token cost attached
  2. MVP — see the fleet ~3–4 wks
    Fleet + Runs views, discovery scanner, run feed via webhooks, Tier-1 cost (Analytics API poller), Docker Compose deploy, docs. Ship publicly; this alone beats clicking through Actions tabs.
    EXIT: a stranger self-hosts it against their org in under 30 minutes
  3. Per-run economics ~3 wks
    scuttledeck/setup@v1 companion action, OTLP ingest endpoint, correlator, PR view with cost-per-review, cost_report reconciliation.
    EXIT: every run shows dollars; every PR shows what Claude's attention cost
  4. Operate — alerts & scale ongoing
    Budget/anomaly/failure alerting to Slack, multi-org, SSO, retention policies. Optional later: hosted SaaS tier (the open-source core stays complete — open-core on hosting, not features).
    EXIT: teams get paged by Scuttledeck before FinOps notices the invoice
10 · Stack

Boring on purpose

Optimized for one-command self-hosting and a tiny operational footprint: a static Go binary for everything server-side, a single database.

DatabasePostgres 16, SQL migrations embedded in the binaryEntities and metrics in one store, migrations applied on boot; ClickHouse only if metric volume ever demands it

LicenseApache-2.0Maximizes adoption in enterprises where the buyers sit; moat is execution, not license
11 · Risks

Open questions & honest risks

RiskSeverityMitigation
Anthropic ships this nativelyHIGHReal possibility (Console already has Claude Code analytics). Defenses: self-hosted + GitHub-native context (PR joins) that Console lacks; multi-source cost truth; move fast. Even so, an OSS tool can win the "my data stays inside" segment.
OTel→run correlation fragilityMEDResource attributes injected by the setup action make the join explicit; heuristic fallback is labeled low-confidence. The spike phase exists to de-risk exactly this.
Discovery scan rate limits on huge orgsMEDETag conditional requests, nightly cadence, push-event triggered rescans of changed workflow files only.
API surface churn (action v2, Analytics API changes)MEDVersion-tolerant YAML matcher (claude-code-action@* and forks); poller payloads parsed tolerantly, alert on drift.
Subscription-OAuth installs have no cost dataLOWDesigned-in state: tokens shown, cost marked "included in subscription."
